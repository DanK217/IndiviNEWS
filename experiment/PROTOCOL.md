# 実験プロトコル: 同一仕様・複数モデル実装比較

同じ仕様書（`experiment/SPEC.md`）を4つのClaudeモデルに新規セッションで渡し、
実装品質と作業効率をQCD観点で比較する。結果は `experiment/RESULTS.md` に記録する。

## 対象モデルと実行順

| 順 | モデル | ブランチ名 | 備考 |
|---|---|---|---|
| 1 | Claude Fable 5 | `exp/fable-5` | **7/7までに実行**（プラン利用期限） |
| 2 | Claude Opus 4.8 | `exp/opus-4-8` | |
| 3 | Claude Sonnet 5 | `exp/sonnet-5` | |
| 4 | Claude Haiku 4.5 | `exp/haiku-4-5` | |

## 各実行の手順

1. mainの**実験基盤コミット**（RESULTS.mdに記録したハッシュ）からブランチを切る:
   `git checkout main && git checkout -b exp/<モデル名>`
2. Claude Codeで**新規セッション**を開始し（前セッションの文脈を持ち込まない）、
   `/model` で対象モデルに切り替える
3. 次のプロンプトを**一字一句同じ内容で**貼り付ける:

   > experiment/SPEC.md を読んで、記載された課題を実装してください。
   > experiment/ 配下の他のファイルは読まない・変更しないでください。
   > 完了したら現在のブランチにコミットしてください。

4. セッション中の人間の振る舞い（公平性のため厳守）:
   - モデルから質問されたら「任せます」とだけ返す
   - 完全に停止した場合のみ最小限の指示を出し、**介入として回数と内容を記録**
   - ヒント・追加要件・感想は一切与えない
   - effort等の設定はデフォルトのまま変更しない
   - 人間も実装セッション中に `experiment/fixtures/` を開かない

## セッション終了後の計測

各ブランチ上で:

```bash
# 1. 指示遵守チェック（experiment/ に触れていないこと。差分があれば違反として記録）
git diff main -- experiment/

# 2. クラスタリング品質の採点（F1）
node experiment/score.js

# 3. ダイジェストが実際に動くか
node scripts/build-digest.js

# 4. ウィジェットが壊れていないか
npm run dev  # → http://localhost:3000 を目視確認
```

記録項目（RESULTS.mdの表へ）:

- **F1 / precision / recall**（score.jsの出力）
- **ターン数**: 人間が送信したメッセージ数（初回プロンプト含む）
- **介入回数**: 軌道修正・停止解除に要した回数と内容
- **初回動作**: 1〜4がモデルの申告どおり一発で通ったか
- **指示遵守**: experiment/への接触有無（セッションログでfixtures閲覧がないかも確認）
- **所要時間**: セッション開始〜完了コミットまで
- **自己検証**: テストや検証手段を自発的に用意したか（SPECは方法を指定していない）

## 目視レビュー（全実装が出そろってから）

採点者バイアスを抑えるため:

- 4ブランチの `src/cluster.js` を無名化（モデル名を伏せてa/b/c/dに）してからレビューする
- 観点: 可読性 / エッジケース処理 / 過剰設計の有無 / 既存コードとの整合
- 欠陥は「動作に影響する」「しない」に分けて数える

## 注意書き（ブログ掲載時にも明記すること)

- 各モデルn=1の**ケーススタディ**であり、統計的なベンチマークではない
- golden set（fixtures）は実装開始前に固定済み（このコミットが証跡）
- 採点スクリプト・golden setはFable 5が作成したため、Fable 5の実装だけは
  「出題者と解答者が同一」になる。ただし実装セッションは新規で行い、
  fixturesの内容は文脈に含まれない
